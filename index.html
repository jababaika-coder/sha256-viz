// 3. ТРИ СВОЙСТВА КРИПТОГРАФИЧЕСКОГО ХЭША
СВОЙСТВО 01
СТОЙКОСТЬ К ПРООБРАЗУ
«Фарш нельзя провернуть назад»
y → x : H(x) = y
✗ НЕВОЗМОЖНО
Зная хэш y, нельзя восстановить x. При вычислении данные намеренно теряются — остатки от деления, побитовые операции.
СВОЙСТВО 02
СТОЙКОСТЬ КО 2-МУ ПРООБРАЗУ
«Нельзя подделать конкретный договор»
x₁ → x₂ : H(x₁) = H(x₂)
✗ НЕВОЗМОЖНО
Зная оригинал x₁, нельзя создать поддельный x₂ с тем же хэшем — даже если искать именно под этот документ.
СВОЙСТВО 03
СТОЙКОСТЬ К КОЛЛИЗИЯМ
«Парадокс дней рождения»
∃ x₁,x₂ : H(x₁) = H(x₂)
✗ НЕВОЗМОЖНО
Нельзя найти любые два документа с одинаковым хэшем. Из-за парадокса дней рождения хэш должен быть ≥ 256 бит.

// 5. ПРИМЕНЕНИЯ В IT
🔐 ХРАНЕНИЕ ПАРОЛЕЙ
KDF(пароль + соль + перец)
соль — уникальна для каждого, хранится в БД. Защита от радужных таблиц.
перец — одна на всех, хранится на сервере (не в БД). Защита при краже базы.
bcrypt / scrypt / Argon2 — намеренно медленные KDF
✍️ ЭЦП / JWT / HMAC
HMAC = H(ключ ‖ данные)
При подписи документа шифруется его хэш, а не сам документ.
JWT: третья часть токена — HMAC от заголовка и payload с секретным ключом.
📊 ХЭШ-ТАБЛИЦЫ
index = H(ключ) % N → O(1)
dict (Python) · HashMap (Java) · объекты в JS
Здесь важна скорость и равномерность, а не криптостойкость.
⛓️ БЛОКЧЕЙН + PROOF OF WORK
H(блок + Nonce) < Target
Каждый блок хранит хэш предыдущего — изменение одной записи рушит всю цепь (лавинный эффект).
Майнинг: вслепую перебираем Nonce, пока хэш не начнётся с N нулей.
